## WHAT THIS PROJECT IS
═══════════════════════════════════════════════════════

A specialist physiotherapy clinical tool for a Lenovo Yoga 
running Fedora 43 in tablet mode. Primary input in bodychart mode is 
pressure-sensitive stylus and touchscreen. Mouse and keyboard 
are secondary. Primary mode in report is keyboard, but it must accept mouse and touchscreen input 100% as well.
Runs on GNOME, with Ptyxis as the terminal.

This is NOT a generic form filler. It is a specialist pain 
assessment system combining:
- Pressure-sensitive body chart drawing with clinical overlays
- A structured specialist assessment with embedded clinical 
  reasoning (Smart et al, ICD-11, biopsychosocial model)
- A context-driven clinical knowledge base that surfaces 
  relevant patterns, checklists and special tests based on 
  what has been drawn and entered
- A live report generator that assembles specialist-grade 
  documentation from structured inputs

The project has two app components and two databases.
All four must be understood before touching anything.

## THREE-LAYER ARCHITECTURE — physio-assessment/
═══════════════════════════════════════════════════════

This is mandatory. Do not mix layers. Ever.

LAYER 1 — models.py (data only)
  Pure state. Pydantic models.
  No UI imports. No logic. No decisions.
  Patient, Session, Barrier, OutcomeMeasure,
  ClinicalPattern, SpecialTestResult, etc.
  All models use: model_config = ConfigDict(extra='ignore')
  This tolerant reader means new body chart JSON fields 
  never crash the assessment app.

LAYER 2 — logic.py (clinical rules only)
  Pure clinical logic. No Textual imports. No UI.
  Reusable outside the UI entirely.
  Contains:
    generate_report_paragraph(section, session_data)
    query_patterns(region, symptom_zones, active_overlay)
    query_tests(pattern_ids)
    score_pattern(pattern_id, features_present)
    suggest_icd11_pathway(duration, mechanism, pain_type)
    suggest_barriers(outcome_scores, pain_type)
    calculate_bmi(height, weight)
    tally_inflammatory_score(features_checked)
    tally_pain_type_features(features_checked)
    interpret_outcome_score(measure_name, score)

LAYER 3 — UI (Textual screens and widgets only)
  Displays state. Captures input. Calls logic layer.
  Never owns clinical logic. Never writes directly to DB.
  Syncs model ↔ UI only.
  Calls storage layer for all persistence.

LAYER 4 — storage.py (persistence only)
  Reads and writes SQLite. No UI. No clinical logic.
  Auto-saves on every field change — no save button.
  Behaves like Obsidian: state always persisted.

  <!-- trigger: -->    → conditional logic
  Tables               → interactive data entry tables

### Alert system — implement exactly:
- Red flag checkbox ticked → mandatory acknowledgement 
  popup, cannot proceed without confirming
- Cauda equina or cord compression positive → URGENT 
  styled alert, requires "action taken" field entry 
  before dismissal
- CSI ≥ 40 → "Suggestive of central sensitisation"
- PCL-5 ≥ 33 → "PTSD likely — document action taken"
- ISI ≥ 10 → "Clinically significant insomnia"
- PCS total ≥ 30 → "High catastrophising — consider 
  psychology referral"
- DASS-21 severe or extremely severe on any subscale 
  → alert with psychiatry referral prompt

### Auto-calculations:
- BMI from height + weight (display only, not editable)
- Inflammatory pain score tally (0–4, auto-count)
- Pain type feature tallies (subjective + examination 
  counted separately)
- Outcome measure score → interpretation label inline

### Barriers checklist → auto-table:
Ticking a barrier auto-populates output table row with 
pre-written strategy text from core/07.
Rows rankable by priority (drag or keyboard up/down).
Custom free-text barriers addable at bottom.

### ICD-11 auto-suggestion:
Suggest most likely pathway based on:
  duration entered + mechanism entered + pain type 
  already classified
Show suggestion with one-line reasoning.
Clinician confirms or overrides.
Never force — suggestion only.

### Report generation:
Each <!-- report-template: --> = one Jinja2 paragraph.
[Preferred name] → replaced with name throughout.
[placeholders] → replaced with entered values.
Centre panel live-updates as fields are completed.
Export: plain text and markdown.

Pre-built templates that auto-populate common

Examples:
  Classic L4/5 disc pattern
  Nociplastic / central sensitisation pattern
  SIJ dysfunction pattern
  Shoulder impingement pattern (when shoulder added)

## CONTEXT ENGINE — logic.py
═══════════════════════════════════════════════════════

The bridge between body chart data and right panel 
content. Pure Python. No UI imports. Fully testable.

query_patterns(region, symptom_zones, active_overlay):
  Queries BodyChartTrigger for matches.
  Scores ClinicalPatterns by trigger match count.
  Returns ranked list of (ClinicalPattern, score).

query_tests(pattern_ids, priority_filter=None):
  Returns SpecialTest list for given patterns.
  Ordered by: essential first, then supporting.
  Includes PatternTest.priority for each.

score_pattern(pattern_id, features_present):
  Takes list of PatternFeature ids confirmed present.
  Returns float 0.0–1.0 confidence score.
  Used to re-rank patterns as assessment progresses.

Right panel query triggers — update right panel when:
  - Region changes
  - Body chart JSON changes (new stroke, overlay change)
  - Any subjective feature checkbox is ticked
  - Any objective feature checkbox is ticked
  - Section navigation changes (show section-relevant 
    content even without body chart data)

## ONGOING PRINCIPLES — apply throughout
═══════════════════════════════════════════════════════

Touch-first. Every interaction works with finger or 
stylus tap before mouse or keyboard is considered.

Maximum vertical space. Minimal chrome. No titlebar.
Left sidebar only. Canvas and content fill everything.

Speed. Used live in clinic. Max 2 taps to complete 
any interaction. No modal dialogs except alerts.

Never lose data. Auto-save to SQLite on every field 
change. No save button anywhere in the app.

Layer discipline. UI never owns logic. Logic never 
owns UI. Storage never owns either. If a function 
needs both a Textual import and a clinical calculation, 
it is in the wrong file — split it.

Tolerant reader. All Pydantic models use 
ConfigDict(extra='ignore'). New body chart JSON 
fields never crash the assessment app.

Schema version discipline. schema_version in every 
session record. clinical_kb_version in kb metadata. 
Migrations required for breaking changes. Never 
break existing records.

Flag, don't guess. Clinical content is precise. If 
a field meaning is ambiguous, a removal is uncertain, 
or a clinical term is unclear — stop and ask. Do not 
interpret clinical content based on assumption.

Knowledge base is clinician-owned. clinical_kb.db 
content is authored by the clinician over time. Seed 
with correct schema and minimal correct content. 
Accuracy matters more than completeness at every stage.
