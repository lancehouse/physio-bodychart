## What was done this session

| Symptom | Pattern | Colour |
|---|---|---|

| Intermittent pain | Dashed line | Red `#E34A4A` |

Files changed: `src/stroke.h` (FillPattern enum), `src/stroke.c` (SYMPTOM_DEFS), `src/canvas.c` (draw_stroke).

---

### 2. Stroke rendering — zoom-invariant line widths

Previously line widths scaled with zoom (zooming in made lines fatter). Now all widths are specified in **screen pixels** and divided by the current scale factor, so a 4px line stays 4px regardless of zoom level. This gives finer detail when zoomed in.

Key formula in `canvas.c`:
```c
double scale = col_base_scale(width, height) * (*cd->p_zoom);
cairo_set_line_width(cr, pixels / scale);
```

---

### 3. Pressure sensitivity

For pain types (solid and dashed), **both width and alpha vary with stylus pressure**:

- Width range: ~1.5px (light touch) → ~7px (full pressure)
- Alpha range: 0.50 (light) → 0.85 (heavy)
- Formula: `base_width × (0.5 + pressure × 2.0) / scale`

> **Hardware note:** Pressure only varies if using a pressure-sensitive active stylus (e.g. Lenovo active pen). Finger touch and capacitive styluses always report ~0.6 pressure and will look uniform.

---

### 4. Performance fix — batched Cairo calls

The first implementation of symbol patterns called `cairo_stroke()` or `cairo_fill()` once per symbol (dozens of calls per stroke object), causing visible lag.

**Fix:** All symbols for a given stroke are now accumulated into a single Cairo path and rendered with one `cairo_stroke()` or `cairo_fill()` call:

- Dots: `cairo_new_sub_path` + `cairo_arc` × N, then one `cairo_fill`
- H-strokes: `cairo_move_to` + `cairo_line_to` × N, then one `cairo_stroke`
- X-marks: same — all 4 arms per X accumulated, one `cairo_stroke`
- Solid: segments batched by pressure band; only flushes when width shifts ≥0.5px

---

### 5. Tabbed sidebar + save / export

The left toolbar was refactored into a **tabbed sidebar** with two tabs:

- **Draw tab** — existing toolbar content (View, Symptom, Tools, Overlay)
- **File tab** — three save/export buttons: SVG, PNG, PDF

Files changed: `src/window.c` (tabbed sidebar structure), `src/session.c` (full implementation), `src/session.h` (added `session_export_pdf`, `session_auto_path`), `src/canvas.c` (added `canvas_render_view`), `src/canvas.h`.

**Save/export behaviour:**
- All three formats auto-name to `BodyChartYYYYMMDDHHMM.xxx`
- Saved to `~/PhysioChart/` (directory created on first save)
- Layout matches the app's quad view: anterior | posterior | lateral L (top-right) | lateral R (bottom-right)
- Export unit = 300px per grid unit → 1500×600 pixel output
- Body outlines use SVG backgrounds if available, Cairo fallback otherwise
- Overlays are **excluded** from export — clean chart only
- After save, the File tab status label shows the saved filename

**Render helper added to canvas.c:**
```c
void canvas_render_view(AppState *app, cairo_t *cr, BodyView view, double w, double h);
```
Renders one body view offscreen at zoom=1, no pan, no overlays. Used by all three export formats.

# Session 2 — 2026-04-30 (continuation)

## What was done

### 1. Sidebar compression to 110px

The left sidebar was over-wide. Fixed at exactly 110px with:
- `gtk_widget_set_hexpand(outer, FALSE)` on the sidebar box
- CSS `min-width: 110px; max-width: 110px` on `#sidebar`
- All button groups converted to `GtkGrid` 2-column homogeneous layout via a shared `make_grid2col()` helper

Result: entire Draw tab content fits without scrolling.

---

### 2. Tick symptom (symptom-free stamp)

New 6th symptom type `SYMPTOM_TICK` / `FILL_TICK` — green checkmarks.

- Single tap places one checkmark; dragging places a chain of checkmarks spaced 14 body units apart
- Single-point strokes kept in `input_end()` for this type only (normally discarded)
- Colour: `#27B344` (green)
- Added `FILL_TICK` case in `draw_stroke()` using `#define TICK_AT(px,py)` macro for batching

Files: `src/stroke.h`, `src/stroke.c`, `src/canvas.c`, `src/input.c`

---

### 3. Note annotation tool

New `TOOL_NOTE` mode. Workflow:

1. User taps "Note ✎" in toolbar → tool switches to TOOL_NOTE
2. Tap on body → `show_note_wizard_cb` fires
3. Centred modal wizard (4 steps, no keyboard needed):
   - Q1: Constant / Intermittent
   - Q2: Superficial / Deep
   - Q3: Average intensity 0–10 (11 buttons)
   - Q4: Worst intensity 0–10 (11 buttons)
4. Wizard auto-closes after Q4; note label placed at tap location

Label format: `(N)C S avg-worst/10` e.g. `(1)C S 3-7/10`

- Up to `MAX_NOTES = 10` annotations per chart
- Labels rendered in screen-space (fixed 11px size regardless of zoom)
- Draggable with TOOL_NOTE active — hit-test checks 130×36px region around label

Key structs: `NoteAnnotation` in `stroke.h`, `WizardData`/`WBtnPair` in `window.c`

Files: `src/stroke.h`, `src/canvas.h`, `src/canvas.c`, `src/window.c`, `src/input.h`

---

### 4. Link tool — fully implemented

"Link" toolbar button opens a modal N×N matrix popup where each cell cycles through:
`blank → → → -/- → blank`

On "Generate Summary":
- All non-neutral pairs stored as `LinkRel[]` array in AppState
- Link summary box placed at body position (12, 378) on the anterior view

**Graphical arrow rendering** (Cairo, screen-space):
- `LINK_YES`: green filled arrowhead with stem (28px wide)
- `LINK_NO`: grey arrow + red diagonal slash crossing the stem
- Summary box is 72px wide, 20px per row, anchored bottom-left at the stored body position

Summary box is draggable across any body view — dragging updates `link_summary_view` to the current drawing area's view.

Popup memory managed via `LinkPopupData` / `on_link_popup_destroy` (same pattern as wizard).

Files: `src/canvas.h` (LinkRel struct, link_rel_count), `src/canvas.c`, `src/window.c`

---

### 5. Cyclable right column

The two right slots in the quad view (`right_slot_views[2]` in AppState) can be independently cycled through all four views by clicking toolbar buttons "L lat ↻" / "R lat ↻".

Implementation:
- `canvas_cycle_right_slot(app, slot)` increments `right_slot_views[slot] % 4`, updates `g_col[2+slot].view` and the header label widget
- Toolbar buttons call this and then `update_toolbar_state()` to refresh button labels
- Export reflects the cycled views via dynamic `ExportSlot` construction in `render_all_views()`
